# RAG по базе знаний

Принимает тикет (обращение клиента), ищет релевантные статьи в базе знаний
(`knowledge_base/`) через Qdrant и формирует пошаговую инструкцию по решению
вопроса моделью на сервере DeepSeek.

## Компоненты

| Файл | Назначение |
|------|------------|
| `kb_loader.py` | Парсинг HTML базы знаний → чанки (заголовок, текст, категория, meta) |
| `embeddings.py` | Локальные эмбеддинги через `fastembed` (мультиязычная e5) |
| `vector_store.py` | Qdrant: коллекция, индексация, поиск |
| `llm.py` | Клиент OpenAI-совместимого сервера DeepSeek (`/v1/models`, `/v1/chat/completions`) |
| `pipeline.py` | `RagPipeline.answer(ticket)` → инструкция + источники |
| `index.py` | CLI: построить индекс |
| `ask.py` | CLI: задать тикет |

## Логика работы

РАГ состоит из двух процессов: **офлайн-индексация** базы знаний и **онлайн-ответ**
на тикет. Эмбеддинги считаются локально (`fastembed`), а текст инструкции генерирует
сервер DeepSeek — он отдаёт только `chat/completions`, поэтому векторизация на него
не завязана.

### 1. Индексация (`python -m rag.index`)

```
knowledge_base/**/*.html
        │  kb_loader.load_chunks()
        ▼
  parse_html_file()                  chunk_text()
  ─────────────────                  ──────────────
  • title  ← <title>/<h1>            • режем тело по абзацам, набирая
  • meta   ← комментарий               ~RAG_CHUNK_SIZE символов;
    helpdeskeddy-kb-meta {...}        • перекрытие RAG_CHUNK_OVERLAP
  • category ← путь папки              переносит «хвост» в начало
  • text   ← очищенный body            следующего чанка (контекст не рвётся);
    (вырезаем <script>/<style>,       • слишком длинный абзац без переносов
     схлопываем пустые строки)          режется скользящим окном.
        │
        ▼  для каждого чанка: text = "<title>\n\n<фрагмент>"
   Embedder.embed_passages()    ← fastembed, модель e5 с префиксом "passage: "
        │  векторы (cosine)
        ▼
   VectorStore.index() → Qdrant upsert
        • id = uuid5("<источник>::<номер_чанка>")  → индексация идемпотентна
        • payload = {text, title, source, category, chunk_index, meta}
```

Почему так:
- **Заголовок добавляется в текст чанка** перед эмбеддингом — короткие фрагменты
  таблиц/команд иначе теряют тему статьи, и поиск по ним хуже.
- **Чанкинг по абзацам с перекрытием** сохраняет смысловые границы (команды, пароли,
  пункты меню обычно лежат в одном абзаце) и не теряет контекст на стыках.
- **Детерминированный id** (`uuid5` от `источник::чанк`) делает повторную индексацию
  обновлением, а не дублированием — можно гонять `index` сколько угодно.

### 2. Ответ на тикет (`python -m rag.ask` / `RagPipeline.answer`)

```
ticket (текст обращения)
        │  Embedder.embed_query()      ← e5 с префиксом "query: "
        ▼
   VectorStore.search() → Qdrant query_points (cosine, top_k=RAG_TOP_K)
        │  релевантные чанки + score
        ▼
   RagPipeline._format_context()       ← нумерованные блоки [N] title/source/text
        │
        ▼
   DeepSeekClient.chat()
        system: RAG_SYSTEM   (роль инженера ТП, формат, «только из контекста»)
        user:   USER_TEMPLATE(ticket, context)
        │
        ▼
   RagAnswer(instruction, sources, model)
        • instruction — текст инструкции от модели
        • sources     — статьи, на которые опирался поиск (title, source, score)
```

Ключевые решения:
- **Анти-галлюцинации.** Системный промпт требует приводить *дословно* команды/
  пароли/серийники из контекста и не выдумывать точные значения, которых там нет.
- **Инструкция выдаётся всегда.** Если прямого решения в базе нет (`solution_found=
  false`) или поиск вообще ничего не вернул — модель всё равно составляет
  *предполагаемую* инструкцию по общим принципам АИИС КУЭ, а оператору показывается
  плашка «ответ предположительный». Так мы не отбиваемся «инструкция не найдена».
  При пустом поиске в контекст подставляется `NO_CONTEXT`, а `solution_found`
  принудительно ставится в `false`.
- **e5-префиксы.** Для моделей семейства e5 запрос и документ кодируются по-разному
  (`query:` / `passage:`) — `Embedder` подставляет их автоматически по имени модели.
- **Имя модели** можно не задавать: `DeepSeekClient` берёт первую из `GET /v1/models`.
- **Источники возвращаются всегда** — оператор видит, на какие статьи опёрся ответ,
  и может перепроверить.

### Тяжёлые импорты — ленивые

`fastembed`, `qdrant_client` и `bs4` импортируются внутри функций, а не на уровне
модуля. Благодаря этому `import rag` и юнит-тесты логики (нарезка, пайплайн на
заглушках, клиент DeepSeek) работают без установки тяжёлых зависимостей.

## Тесты

```bash
pip install pytest
python -m pytest            # из корня проекта
```

- `tests/test_kb_loader.py` — нарезка `chunk_text` (граничные случаи) и парсинг
  HTML (`parse_html_file`/`load_chunks`; требуют `beautifulsoup4`).
- `tests/test_llm.py` — `DeepSeekClient` против `httpx.MockTransport` (без сети):
  список моделей, автовыбор модели, payload и парсинг ответа, заголовок авторизации.
- `tests/test_pipeline.py` — `RagPipeline` на заглушках хранилища и LLM: сборка
  контекста, источники, ветка «нет совпадений» (LLM не вызывается), пустой тикет.

Тесты не требуют сети, Qdrant и DeepSeek-сервера.

## Настройка

1. Зависимости: `pip install -r requirements.txt`
2. Поднять Qdrant, например: `docker run -p 6333:6333 qdrant/qdrant`
3. Прописать в `.env` (см. `.env.example`):
   - `DEEPSEEK_BASE_URL=http://chatbot.lar.tech:8081/v1`
   - `QDRANT_URL=http://localhost:6333`
   - при необходимости `DEEPSEEK_MODEL`, `EMBED_MODEL`

> Если Qdrant-сервера нет, можно указать `QDRANT_URL=:memory:` (в памяти, без
> персистентности) или путь к папке для локального хранилища на диске.

## Использование

```bash
# 1. построить индекс из knowledge_base
python -m rag.index --recreate

# 2. задать тикет
python -m rag.ask "Меркурий 206 не выходит на связь, что делать?"
echo "текст тикета" | python -m rag.ask --json
```

Программно:

```python
from rag import RagPipeline

answer = RagPipeline().answer("ПУ залип в реджойне")
print(answer.instruction)
print(answer.sources)   # [{title, source, category, score}, ...]
```

## Замечания

- Эмбеддинги считаются локально (`fastembed`) — сервер DeepSeek отдаёт только chat,
  поэтому векторизация не зависит от него. Модель `intfloat/multilingual-e5-large`
  скачивается при первом запуске.
- Имя модели DeepSeek можно не задавать — берётся первая из `GET /v1/models`.
- Индексация идемпотентна: id точки детерминирован по `источник::номер_чанка`,
  повторный запуск обновляет, а не дублирует.
